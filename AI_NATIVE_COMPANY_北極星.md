## 一、什麼是 AI 原生公司

**不是「用 AI 的公司」**(把 ChatGPT 當工具,那是貼皮)——而是:

> **公司的營運本身,跑在一個「會自己觀察、自己提改進、由人批准、用程式當法律」的閉環上。**

明確**拒絕玩具級展示**(查個資料表就號稱 AI 公司)。

### 讓它「原生」而非「貼皮」的三個特徵
1. **一切留痕、不可竄改(Legibility)** — 每件事寫進 append-only 事實庫,AI 能讀自己完整的過去 → 才學得起來。
2. **系統自己提改進** — 發現答錯/失敗 → 提出修正(規則 / prompt / 資料 / 程式),不是工程師每次手動補。
3. **人是閘門、Code 是法律** — 沙盒驗證 + 人批准 + 可回滾才上線;路由與決策是**確定性程式**,LLM 只提案、不主宰流程。

## 二、架構分層(怎麼做才對)

```
L0 真相層 ── 本機 PostgreSQL,append-only、不可竄改、單一事實來源
            (對話/訂單/行程/事件全留痕)
                    │
L1 知識管線 ── 確定性清洗 → 去識別 → 標記 → canonical 知識
            可更新 / 可調整 / 抓最新(對話 + 行程,多來源)
                    │
L2 路由層 ── Code is Law:Python 規則決定「怎麼處理」
            (查訂單?查行程?回覆?升級真人?)
                    │
L3 AI 輔助層 ── AI「提案」:RAG 草擬回覆、提改進建議
            **絕不自動回客人、不自動改生產**
                    │
L4 閉環學習 ── 提案 → 沙盒驗證 → 人批准 → 安全上線/可回滾 → 留痕 → 再進化
            (含「AI 建議 vs 人修正 vs 理由」回饋)
                    │
L5 人類閘門 ── 人做判斷與批准(Telegram 批准 / 客服改稿送出)
```

---

## 三、橫貫所有層的鐵則

- **Code is Law**:確定性控制流;LLM 是提案者與苦工,**不是 orchestrator**。
- **本機為真相 + 可攜**:真相與 AI 在地(省算力、可控、PII 主權);用標準元件(PostgreSQL / Qdrant / 檔案),連線走設定不寫死 → 想搬雲端只是搬資料,不是重寫。資料位置 ≠ 運算位置。
- **不可逆 / 對外動作 → 人先批准**;**AI 不自動回客人、不自動改生產**。
- **一切留痕(legibility)**:不可竄改稽核 → 才可學、可回滾、可究責。
- **異地備份**:本機是真相 ≠ 只有一份(冷備份到 QNAP / 雲端 object storage)。

## 五、反面教材(最容易做錯的)

- 讓 LLM 自由決定流程(當 orchestrator)→ 破壞 Code is Law、不可預測。
- AI 直接自動回客人 / 直接改生產 → 沒有人閘門。
- 真相散在各家雲端 SaaS、沒留痕 → 學不起來、被廠商綁死、成本失控。
- 停在玩具 demo(炫技但沒閉環)。

## 一句收尾

**AI 原生 = 把公司營運變成「可讀、可自我改進、人在閘門、程式為法」的閉環;AI 負責提案與苦工,人升級到判斷與批准。**

我們是「本機真相庫 + 確定性知識管線 + 規則路由 + 建議模式 + 閉環學習」一塊一塊長出來,最終串成一個**會自己變好的公司**。
